## 摘要
微服务应用越来越多地部署在云-边缘环境中，异构节点和时变的节点间延迟放大了放置决策的影响。同时，这些应用面临非平稳流量、执行不同调用图的根请求操作组合的变化，以及决定网络延迟和队列如何传播到端到端（E2E）性能的异构通信模式。现有的自动缩放器和网络感知调度器通常只处理这些动态的一部分，导致计算瓶颈或跨节点延迟膨胀，从而违反SLO。我们提出ADASCALE，一种自适应框架，在这种多维动态下联合缩放和放置微服务副本。ADASCALE实现了一个监控-分析-规划-执行（MAPE）循环，从分布式追踪和服务网格指标中提取每个边缘和每个服务的需求，识别混合工作负载下最关键的根操作，计算SLO感知的副本目标，然后根据当前节点间延迟矩阵放置副本以最小化需求加权延迟目标。为了快速响应网络扰动，ADASCALE触发一个反应式放置循环，而稳态自动缩放循环处理需求变化。我们在使用DeathStarBench社交网络应用的云-边缘Kubernetes集群上评估ADASCALE，该应用在变化负载和工作负载组合下包含三种根操作。在各种场景中，ADASCALE始终满足SLO目标，并改善了延迟和吞吐量：与NetMARKS_Scale相比，平均响应时间降低了最多1.56倍、1.93倍和1.34倍（针对compose-post、read-home-timeline和read-user-timeline），吞吐量分别提高了最多2.16倍、1.32倍和1.36倍。

## Abstract
Microservice applications are increasingly deployed across cloud--edge environments, where heterogeneous nodes and time-varying inter-node delays amplify the impact of placement decisions. At the same time, these applications face non-stationary traffic, shifts in the mix of root request operations that exercise different call graphs, and heterogeneous communication modes that determine how network latency and queuing propagate to end-to-end (E2E) performance. Existing autoscalers and network-aware schedulers typically handle only a subset of these dynamics, leading to either compute bottlenecks or inflated cross-node latency and thus SLO violations.   We propose ADASCALE, an adaptive framework that jointly scales and places microservice replicas under such multi-dimensional dynamics. ADASCALE implements a Monitor--Analyzer--Planner--Executor (MAPE) loop that extracts per-edge and per-service demand from distributed traces and service-mesh metrics, identifies the most critical root operation under a mixed workload, computes SLO-aware replica targets, and then places replicas to minimize a demand-weighted latency objective given the current inter-node latency matrix. To react quickly to networking perturbations, ADASCALE triggers a reactive placement loop, while a steady-state autoscaling loop handles demand shifts.   We evaluate ADASCALE on a cloud--edge Kubernetes cluster using the DeathStarBench Social Network application with three root operations under varying load and workload mixes. Across scenarios, ADASCALE consistently meets SLO targets and improves both latency and throughput: compared with NetMARKS_Scale, it achieves up to 1.56x, 1.93x, and 1.34x lower average response time (for compose-post, read-home-timeline, and read-user-timeline) and up to 2.16x, 1.32x, and 1.36x higher throughput, respectively.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究背景**：微服务架构广泛应用于云-边缘环境，但面临多维动态挑战：
  - **节点异构**：云-边缘节点计算、网络性能差异大。
  - **时变网络延迟**：跨节点通信延迟随时间波动，影响端到端（E2E）性能。
  - **非平稳流量**：用户请求量随时间变化，具有突发性。
  - **混合根操作**：不同请求类型（如写帖、读主页、读用户时间线）对应不同调用图，且比例动态变化。
  - **异构通信模式**：服务间通信模式（如HTTP、gRPC、DB、MQ等）对网络延迟和队列的敏感性不同。
- **现有方法不足**：
  - Kubernetes HPA只基于CPU/内存利用率缩放，忽略网络状态和操作混合。
  - 网络感知调度器（如NetMARKS）虽考虑网络，但假设固定副本预算，不处理请求混合变化。
  - 调用图分析工具可定位瓶颈，但未设计为驱动联合缩放和放置决策。
- **核心问题**：如何统一、自适应地处理多维动态，实现零SLO违例，降低延迟并提升吞吐量。

## 2. 方法论

### 核心思想
- **联合缩放和放置**：同时决定每个服务的副本数（缩放）和副本的节点分布（放置），以最小化资源成本与需求加权网络延迟之和，满足E2E和单个服务的SLO约束。
- **两时间尺度自适应**：
  - **稳态自动缩放循环**：处理需求变化和操作混合变化，更新副本目标。
  - **反应式放置循环**：快速响应网络扰动（如延迟变化），重新放置副本。
- **MAPE控制循环**：监控（Monitor）→分析（Analyzer）→规划（Planner）→执行（Executor）。

### 关键技术细节

#### 系统模型与状态表示
- 服务集合 \(S\)，节点集合 \(N\)，副本分配矩阵 \(A(t) = [A_{s,n}(t)]\)，表示服务 \(s\) 在节点 \(n\) 上的副本数。
- 根请求分为 \(K\) 类，每类对应调用图 \(G_k\)，具有比例 \(\pi_k(t)\)。
- 通过分布式追踪（Jaeger）和服务网格（Istio）指标提取每个边（服务对）的统计量：调用率 \(r_e(t)\)、平均服务时间 \(w_e(t)\)、字节率 \(b_e(t)\)。
- 服务级需求：入/出调用率、平均工作、CPU需求等。

#### 优化目标（公式5-9）
- 最小化总成本：资源成本 + 需求加权网络延迟。
- 约束：节点容量、E2E SLO（q分位数）、服务级SLO。
- 分解为两步：先通过需求驱动计算副本数，再贪心放置。

#### 缩放算法（Algorithm 1：Service Scaler）
- 基于多根操作的关键性（SLO违例程度 × 比例）选择最危险的根操作。
- 对被选中的根操作调用的服务，计算压力比、追踪关键性、需求份额，得到综合分数。
- 输出建议的缩放动作（up/down/hold），并限制每轮仅top-K服务可扩容。
- 同时输出加权边集合 \(E(t)\)（权重与调用率 \(r_{(u,v)}\) 成正比）供放置使用。

#### 放置算法（Algorithm 2：Service Placer）
- 输入：当前分配、副本目标、延迟矩阵、加权边集合、节点容量。
- 使用贪心策略：逐个添加或移除副本，每次选择使成本 \(Cost(A) = \text{LatCost} + \text{CapPenalty}\) 最小的节点。
- LatCost基于加权集合和副本平均边缘延迟。

#### 执行器
- 通过Kubernetes原生机制（如更新Deployment副本数、节点亲和性约束）执行计划，支持并行滚动更新。
