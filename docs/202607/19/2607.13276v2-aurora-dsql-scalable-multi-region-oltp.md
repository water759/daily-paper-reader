## 摘要
Aurora DSQL是一种无服务器SQL数据库，专为具有多区域主动-主动能力的云规模事务处理而设计。它基于分离式架构构建，将计算、存储和事务协调分离为独立的、水平可扩展的服务。查询处理器在Firecracker微虚拟机中运行，执行兼容PostgreSQL的SQL，而无需保留本地状态。该系统使用带有精确时间戳的多版本并发控制来实现无协调读取，并使用乐观并发控制处理写入，通过分布式裁决器和日志复制系统将协调推迟到提交时。这种方式通过仅在提交时（而非单个语句）进行协调，最大限度地减少了跨区域延迟。DSQL支持从零到每秒数百万笔事务的弹性扩展，同时提供强一致性、ACID事务以及在可用区或区域故障期间的持续可用性。

## Abstract
Aurora DSQL is a serverless SQL database designed for cloud-scale transaction processing with multi-region active-active capabilities. Built on a disaggregated architecture, DSQL separates compute, storage, and transaction coordination into independent, horizontally scalable services. Query processors run in Firecracker MicroVMs executing PostgreSQL-compatible SQL without local state. The system uses multiversion concurrency control with precision timestamps for coordination-free reads and optimistic concurrency control for writes, deferring coordination to commit time through distributed adjudicators and the Journal replication system. This minimizes cross-region latency by requiring coordination only during commits, not individual statements. DSQL enables elastic scaling from zero to millions of transactions per second while providing strong consistency, ACID transactions, and continuous availability during availability zone or region failures.

# Aurora DSQL: Scalable, Multi-Region OLTP 论文深度总结

## 1. 核心问题与整体含义（研究动机与背景）

- **背景**：AWS 虽已拥有 DynamoDB、Aurora 等数据库，但缺少一款水平可扩展、支持多区域主动-主动架构、且面向 OLTP 的 SQL 数据库。客户需求包括：无服务器管理、熟悉 SQL 接口、从零到数百万 TPS 的弹性伸缩、多区域强一致性、连续可用性。
- **核心问题**：传统云数据库在多区域部署时面临扩展性瓶颈（单主架构）、跨区域协调延迟高（悲观锁导致多次往返）、以及可用性不足（区域故障需人工干预）。需要一种既能提供强一致性又能最小化跨区域通信的架构。
- **整体意义**：通过创新设计，让开发者能够使用简单的单区域或多区域架构构建全球级业务，无需关心底层的扩展、容错和运维问题，真正实现“数据库首选”。

## 2. 方法论：核心思想与关键技术细节

### 2.1 分离式架构
- **组件解耦**：系统分为 Query Processor（QP）、Storage Node、Adjudicator（裁决器）、Journal（日志服务）、Crossbar（数据分发）五大独立组件，每个组件均可独立水平扩展。
- **无状态 QP**：每个 QP 运行在 Firecracker 微虚拟机中，执行 PostgreSQL 兼容 SQL，但无本地持久状态；写入仅缓存在本地 QP，提交时才进入协调路径。

### 2.2 读取路径：基于精度时间戳的 MVCC
- **事务开始时间**：使用 EC2 精度时钟获取微秒级 `τ_start`，作为事务的 snapshot 时间。
- **无协调读取**：QP 向存储节点请求数据时携带 `τ_start`，存储节点返回该时间点之前的最新已提交版本。读取永远不需要与主副本或锁服务器通信，可在同一可用区内的任何副本执行。
- **读提交优化**：只读事务的 COMMIT 为无操作，QP 直接丢弃事务状态，不产生额外通信。

### 2.3 写入与提交：乐观并发控制（OCC）
- **写入本地化**：UPDATE/INSERT 操作在 QP 中记录变更，不立即写入存储。
- **提交阶段协调**：提交时 QP 将写入键集发送给涉及的 Adjudicator。裁决器检测写-写冲突（检查 `τ_start` 到 `τ_commit` 之间是否有其他事务修改了相同键）。若无冲突，则选择主裁决器将事务写入 Journal，并确定 `τ_commit`。
- **跨裁决器提交**：使用类似两阶段提交但非原子性投票的协议：所有涉及的裁决器投票，主裁决器收集后写入自己的 Journal（唯一写入点），无需跨 Journal 原子提交，避免传统两阶段提交的协调者故障问题。
- **Journal 与复制**：Journal 是一个有序原子数据流，单区域时写入两个可用区，多区域时写入至少两个区域。Crossbar 从 Journal 读取变更并分发到对应存储分片。

### 2.4 时钟与一致性
- **时钟精度保证**：AWS 提供微秒级同步时钟，`τ_start` 和 `τ_commit` 均基于此。时钟偏移超出预期会导致失去线性一致性（退化为只保证 snapshot isolation），但不会丢失隔离性、持久性或原子性。
- **强一致性选择**：系统选择强 snapshot isolation（即 snapshot isolation + linearizability），仅在网络分区时牺牲可用性（保持 PC/EC 中的 PC）。

## 3. 实验设计与评估

### 3.1 实验场景与数据集
- **微基准测试**：使用简单主键 SELECT、UPDATE、COMMIT 操作，分别在单区域（us-east-2, SR）和两区域（us-east-1 + us-east-2 + 见证节点 us-west-2）设置下测量 p99 延迟。
- **对比系统**：竞争对手 A（悲观锁分布式 SQL 系统），对比不同语句数量下的平均延迟。
- **仿真实验**：使用 TPC-C 风格 benchmark 进行事件驱动数值仿真，比较单区域、近区域（Virginia-Ohio）、远区域（Virginia-Oregon）三种配置下的良好吞吐量（goodput）。

### 3.2 关键性能数据
- **读取延迟**：多区域设置下主键 SELECT p99 约 2ms（同一可用区），UPDATE p99 约 3ms。
- **提交延迟**：多区域提交 p99 约 30ms（远快于最远区域 RTT 63ms，得益于 2/3 仲裁提交）；单区域提交 p99 约 7.4ms。
- **对比优势**：与竞争对手 A 相比，DSQL 的事务延迟不随语句数线性增长（因为无需每语句协调），而竞争对手 A 延迟随语句数线性上升（平均延迟增长约 30 倍）。

### 3.3 仿真结果
- TPC-C 仿真显示：跨区域延迟对良好吞吐量影响有限，整体系统能够良好扩展；近区域配置与单区域接近，远区域稍有下降但仍在可接受范围。

## 6. 主要结论与发现

- **弹性与性能**：DSQL 实现了从零到数百万 TPS 的弹性扩展，仅需在提交时进行一次跨区域协调，显著降低了多区域事务延迟。

## 7. 优点

### 方法亮点
- **协调最小化**：读取完全无协调，写入仅提交阶段需一次跨区域往返，比悲观锁系统减少大量网络开销。
- **无锁设计**：OCC 避免锁等待、死锁和客户端阻塞，提升系统稳定性和尾部延迟。
- **独立扩展**：计算、存储、协调可独立伸缩，控制平面基于热度预测自动 split/merge，无需停机。
