## 摘要
模型托管中心（如Hugging Face）容易受到供应链攻击，这类攻击能在受信任的用户环境中实现远程代码执行。攻击者通常通过模型中心分发恶意的预训练机器学习模型（PTMs）。在本文中，我们提出了一种名为SHADOWPICKLE的新颖攻击方法，针对PTMs和模型中心。SHADOWPICKLE包含三种（3）隐蔽的Pickle反序列化攻击，能够实现恶意行为并规避最先进的（SOTA）模型扫描器。这些攻击利用Pickle虚拟机（VM）的外部模块导入机制，在反序列化期间执行恶意负载。此外，我们提供了PICKLEBENCH，这是一个动态且可扩展的基准测试，用于自动将SHADOWPICKLE注入任意良性PTM模型。我们的评估表明，SHADOWPICKLE能够规避十个SOTA扫描器和四个模型中心。SHADOWPICKLE（重写）在扫描器中的规避率为63%，比现有攻击的规避率高出高达50%。此外，PICKLEBENCH比三个SOTA基准测试的挑战性高出高达25.6%。最后，我们提供了缓解我们攻击并提高现有扫描器有效性的安全建议。我们的发现揭示了现有PTM扫描器的局限性，并指出了改进方向。

## Abstract
Model hosting hubs (e.g., Hugging Face) are vulnerable to supply chain attacks that enable remote code execution on trusted user environments. Attackers often distribute malicious Pre-trained ML models (PTMs) via model hubs. In this paper, we present novel attacks against PTMs and model hubs called SHADOWPICKLE. SHADOWPICKLE includes three (3) stealthy pickle deserialization attacks that enable malicious behaviors and evade state-of-the-art (SOTA) model scanners. These attacks leverage the external module import mechanism of the Pickle Virtual Machine (VM) to execute malicious payloads during deserialization. Additionally, we provide PICKLEBENCH, a dynamic and extensible benchmark for automatically injecting SHADOWPICKLE into arbitrary benign PTM models. Our evaluation shows that SHADOWPICKLE evades ten SOTA scanners, and four model hubs. SHADOWPICKLE (Overwritten) has a 63% evasion rate across scanners, and up to 50% higher evasion rates than existing attacks. Besides, PICKLEBENCH is up to 25.6% more challenging than three SOTA benchmarks. Finally, we provide security recommendations for mitigating our attacks and improving the effectiveness of existing scanners. Our findings highlight the limitations of existing PTM scanners and suggest directions for improvements.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：预训练模型（PTM）托管中心（如 Hugging Face）已成为供应链攻击的目标。攻击者通过上传恶意 Pickle 格式模型，利用 Pickle 反序列化漏洞（CWE-502）在用户环境中执行任意代码。当前最先进（SOTA）的模型扫描器（静态、动态、模型加载环境、闭源扫描器）依赖非穷尽的黑/白名单，已无法检测新型隐蔽攻击。
- **意义**：揭露现有扫描器的根本缺陷——仅修补症状而非解决通用类漏洞（如 Pickle VM 的 GLOBAL 导入机制），并证明模型中心依赖的扫描器、社区维护的包索引（PyPI）以及默认的 PyTorch 加载环境（WeightsOnlyUnpickler）均存在严重安全不足。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
利用 Pickle VM 在反序列化时通过 `GLOBAL` 等操作码导入模块执行代码的机制，通过**合成自定义模块**（而非依赖已有库中的 gadget）来绕过扫描器的黑/白名单。三种攻击变体均保持 Pickle 文件有效且可反序列化。

### 关键技术细节

- **PyPI 注入攻击**：将恶意负载（如反向 shell）注入 PTM，依赖一个 PyPI 上的第三方库（如 `raft`）执行任意代码。模型中包含 `GLOBAL raft.run` 等操作码。用户需安装该库（通过 `requirements.txt`），并使用 `weights_only=False` 加载模型。
- **外部模块攻击**：将恶意外部 Python 模块（如 `external.py`，包含 `dangerous_func` 函数）与 PTM 一起上传。模型通过 `GLOBAL external.dangerous_func` 调用该模块执行代码。用户需将外部模块置于同一目录或加入 `sys.path`。
- **重写模块攻击**：攻击者上传一个**重写 Python 标准库模块**（如 `collections.OrderedDict`）到 PyPI，添加恶意功能（例如在 `__new__` 中添加 `exec`）。用户在安装该库后，模型中的 `GLOBAL collections.OrderedDict` 会加载重写的模块，从而执行任意代码。该攻击**完全绕过 PyTorch 的 WeightsOnlyUnpickler**，因为 `OrderedDict` 在其白名单中。

### 算法流程（PICKLEBENCH）
1. **爬取**：从 Hugging Face 下载 1000 个最受欢迎、未被标记为恶意且为 Pickle 格式的良性 PTM。
2. **负载生成**：从 PayloadsAllTheThings 和 revshells.com 收集 20 种真实负载（侦察、反向 shell 等），分别按三种攻击类型生成恶意 Pickle 文件。
3. **注入**：随机选择负载 Pickle，调整其内存地址（避免与原始模型堆栈冲突），注入到良性 PTM 中，并验证模型可加载且负载可执行。
4. **扫描**：使用 SOTA 扫描器对每个注入模型进行检测，记录结果。

- **充分性**：覆盖了主流扫描器类型（静态/动态/模型加载环境/闭源）、现有已知攻击、多种攻击变体、先进绕过技术。使用了大量真实 PTM（3000 个），并注入合成恶意模型，同时验证了与真实恶意基准的一致性。

## 6. 主要结论与发现

- **ShadowPickle 高效规避扫描器**：三种攻击在所有扫描器上的平均**未检测率为 48.5%**（7280/15000）。重写模块攻击的**逃逸率最高达 63%**（仅 39/1000 被 WeightsOnlyUnpickler 检测，且这 39 个因额外操作码问题而非真正检测）。
- **对比现有攻击**：ShadowPickle 的平均逃逸率比 Stacked Pickles、Library Import、PICKLE CLOAK 等**高出 50%**（以重写攻击 vs. 最佳现有攻击 PICKLE CLOAK Module-Surface 为例，逃逸率高 36%）。
- **现有扫描器性能严重不足**：PickleScan 和 ModelScan 对 ShadowPickle 的**召回率为 0**（完全无法检测）。Fickling 假阳性极高（2834/3000 良性模型被误报）。ModelTracer 召回率 0.8937，但易被读取类系统调用绕过。
- **突破 PyTorch 安全机制**：重写模块攻击**首次成功绕过 PyTorch 默认的 WeightsOnlyUnpickler**（设置了 weights_only=True 时仍被绕过）。
- **PICKLEBENCH 更具挑战性**：比 MAL HUG、PickleBall、PICKLE CLOAK 等基准**难检测 25.6%**（平均）。
- **防御补丁有效**：通过添加环境完整性检查（检查 sys.modules 中白名单库是否来自非标准路径），可将 WeightsOnlyUnpickler 的 F1 提升 **19%**（从 0.8007 到 0.9911），Fickling 的 F1 提升 **5%**（从 0.6791 到 0.7165），假阳性降低 **16%**。

## 7. 优点

- **方法新颖性**：提出**重写白名单模块**的攻击思想，首次利用 Python 标准库路径覆盖绕过白名单，攻击点直指扫描器安全假设的根基（环境未被篡改）。
- **基准可扩展性**：PICKLEBENCH 提供自动注入脚本，支持添加新的 PyPI 库和负载，便于未来评估扫描器更新。同时包含黑名单和白名单两种评估维度。
- **实验设计全面**：同时涵盖开源和闭源扫描器、多种攻击类型、先进绕过技术、真实恶意基准对比；并对每个注入模型进行格式和负载有效性验证（确保不是无效 Pickle，与 PICKLE CLOAK 不同）。
- **防御贡献**：不仅指出问题，还提供了可落地的防御补丁（环境检查）和安全建议（使用 gadget 查找工具、扫描 requirements.txt 等）。
- **代码开源**：提供完整代码和测试数据集，便于复现。

## 8. 不足与局限

- **实验覆盖局限**：仅使用 Hugging Face 的文本生成任务 PTM（任务标签单一），未涉及图像、多模态等模型。闭源扫描器测试仅使用少量示例模型（每个攻击一个），而非大规模上传验证（因安全限制）。
- **攻击前提条件**：PyPI 注入和外部模块攻击**要求用户手动安装第三方库或下载外部文件**（尽管实际情况中很多模型确实需要）。重写模块攻击则需要用户安装重写的 PyPI 包。
- **合成性**：恶意模型均为**人工注入**，可能未完全反映真实世界中攻击者的混淆或变异方式。作者虽用真实数据集验证了基准一致性，但未直接使用真实未被检测的恶意模型。
- **扫描器版本时效**：实验时使用的扫描器版本（如表 XII）可能已更新，部分攻击在新版本中可能被检测（但补丁实验表明整体问题仍在）。
- **算力与部署细节缺失**：未比较扫描器的运行时开销或假阳性对用户体验的影响。未在真实用户环境中测试攻击危害（仅安全实验室环境）。
- **泛化性有限**：重写模块攻击依赖特定的 Python 标准库（`OrderedDict`、`xxsubtype`），虽然作者演示了另一个库，但未覆盖所有可能。从 PyPI 到模型加载的供应链链条实际可能更复杂（如依赖版本冲突）。
