## 摘要
自我托管的AI智能体通过读写自身的内存和配置文件来运行。智能体可能因其自身状态的损坏而受到损害——这种损害是通过合法的操作系统系统调用实现的。我们将这类威胁称为自我状态攻击。在本文中，我们研究了操作系统对此类攻击的韧性。形式上，我们刻画了一个四轴攻击空间（目标、机制、粒度、时间性）；探讨了预防、检测和恢复的结构性限制；并引入了基于工作负载的可检测性观点。为了实例化该框架，我们从运行在不同工作负载配置文件下的代表性自我托管智能体中收集实时活动轨迹，并将攻击空间实现为一个23单元格矩阵，对真实自我状态文件执行43个具体操作，并注入到这些轨迹中。然后我们评估了经典的和基于工作负载的防御策略。实验结果表明，分层防御堆栈（指令层和配置层的访问控制预防，内存层基于工作负载的检测，以及定期备份恢复）在大多数攻击单元格上是有效的，而少量残余攻击面在操作系统层面仍然在结构上不可区分。这些发现表明，针对新建立的自我状态攻击类别，需要重新考虑操作系统层面的防御，可能为该领域开辟新的研究方向。

## Abstract
Self-hosted AI agents read and write their own memory and configuration files to function. An agent may get compromised via corruption of its own state -- a compromise realized via legitimate OS system call invocation. We refer to this class of threats as self-state attacks. In this paper, we investigate the OS resilience to this class of attacks. Formally, we characterize a four-axis attack space (Target, Mechanism, Granularity, Temporal); investigate the structural limits of prevention, detection, and recovery; and introduce a workload-conditioned view of detectability. To instantiate the framework, we collect live activity traces from a representative self-hosted agent running across distinct workload profiles, and realize the attack space as a 23-cell matrix, 43 concrete operations on real self-state files, and injected into those traces. We then evaluate both canonical and workload-conditioned defense strategies. The empirical results show that a layered defense stack (access-control prevention on the instruction and configuration layers, workload-conditioned detection on the memory layer, and periodic backup for recovery) is effective on most attack cells while a small residual attack surface remains structurally indistinguishable at the OS level. These findings suggest that against the newly established class of self-state attacks, OS-level defense needs to be reconsidered, potentially opening new research directions in the field.

### 论文主要结论与发现

1. 自我状态攻击在OS层存在结构性的不可区分残差：在推荐的L3预防 + B2检测 + 15事件备份的组合下，23个单元格中4个（Mem-M1-G3, Mem-M1-G4, Mem-M2-G3, Mem-M2-G4）在至少一个工作负载下TPR<50%，属于不可避免的I类。它们集中在“记忆”层的小规模修改或每日日志追加中，特征与合法写入分布完全重叠。
2. 工作负载条件化检测（B2）比全局盲检测（B1）更精确：将2个I类细胞提升为C类，同时暴露了2个原先被全局基线隐藏的C类（真实漏洞），并显著提高了C类细胞的平均TPR（+0.37），且FPR可控（W3上约8%）。
3. 预防策略L3（锁定指令和配置层，保留记忆层可写）是推荐操作系统，在不严重影响工作负载的前提下阻挡了61%的攻击细胞。
4. 恢复在检测触发后能100%修复V/C/I类攻击，但代价是损失备份区间内的合法事件（约15%），且备份目录本身可能成为攻击目标。
5. 时间逃逸仅在爆发对齐模式下对极个别细胞有效（1个C→I），不影响总体格局。

### 优点

- **形式化贡献**：首次给自我状态攻击赋予严谨的四维结构，并指出防御的每个维度都有根本局限。  
- **工作负载条件化视角**：提出检测效果依赖部署场景，而非通用阈值，具有实用指导意义。  
- **系统性评估**：覆盖了预防、检测、恢复三大OS防御面，并量化了各层面的安全/功能平衡。  
- **真实轨迹驱动**：攻击注入在真实代理事件序列中进行，而非合成场景，增强了结论的现实性。  
- **开源可复现**：提供了完整实验平台和攻击套件，便于后续研究。

### 不足与局限

- **实验覆盖有限**：仅针对OpenClaw架构和Gemini 3 Flash模型，未评估其他自托管代理（如Claude Code、Open Interpreter）或其他LLM。不同代理的写分布可能不同，V/C/I分类可能迁移。  
- **工作负载多样性**：虽然四个配置文件涵盖不同场景，但W1/W2/W3/W4的写分布可能仍不足以代表所有部署。例如，强调自我反思的代理会更频繁写指令文件，可能使一些V类细胞退化为C类。  
- **检测器特征空间局限于Per-event**：论文明确指出未利用跨事件的轨迹级或内容级特征。未来更高层的检测（如活动序列异常）可能减少I类细胞，但属于结构上限之外的工作。  
- **内容无感知**：OS层天然无法感知写入内容的语义恶意，因此对微小的“事实翻转”无法检测。这属于根本局限，但论文已清楚指出。  
- **资源消耗未量化**：未报告检测/恢复的运行时开销，实际部署时inotify监控和实时统计计算可能存在性能影响。  
- **恢复条件**：修复依赖于检测在备份间隔内触发，且恢复仅还原文件，未考虑跨层传播效果（如内存污染后的指令修改），实际修复复杂度可能更高。
