## Abstract 

Large-scale distributed file systems must provision resources for peak demand, yet file access patterns fluctuate significantly, leaving substantial capacity idle during off-peak periods. Existing scaling mechanisms operate at the granularity of entire servers and take minutes to hours, making them unable to track the rapid, fine-grained load variations that file systems commonly experience. Serverless computing, with its millisecond-granularity elasticity and pay-per-use pricing, offers a compelling alternative. We present 

SLFS , the first distributed file system built with serverless functions for both data and metadata operations. SLFS implements file ser-vices on top of key-value stores, keeping function operations simple and short, and introduces a novel multi-threaded, short-lived server design that overcomes the cold-start problem while maintaining low cost. A policy-enforcing coordinator efficiently maps files to function instances, scales the system elastically, and controls func-tion lifetimes to balance performance and cost. SLFS can flexibly run on diverse storage backends—from cloud-native services like S3 to user-managed key-value stores—enabling configurable cost-performance trade-offs. Our evaluation shows that SLFS mitigates cold starts by 580 × compared to the base serverless design and outperforms 𝜆 FS , EFS, and Ceph at up to 63%, 68%, and 63% lower cost, respectively. 

## 1 Introduction 

Large-scale distributed file systems are composed of hundreds of server nodes where some nodes handle file operations while others focus on storing data [ 32 , 49 , 57 , 65 , 78 , 83 ]. In practice, file system load fluctuates significantly: not all files are actively accessed at any given time, and access patterns shift unpredictably across time of day, workload phase, and tenant activity. Yet, to meet peak de-mand and maintain acceptable latency, operators must provision resources for worst-case load, leaving substantial capacity idle dur-ing off-peak periods. This over-provisioning translates directly into wasted cost. Existing distributed file systems offer scaling mech-anisms, but these operate at the granularity of entire servers or storage nodes—adding or removing a node involves data migra-tion, rebalancing, and reconfiguration that can take minutes to hours [ 7, 83 ]. Such coarse-grained scaling cannot track the rapid, fine-grained load variations that file systems commonly experience. 

> ∗Both authors contributed equally to this research.

Serverless computing offers a compelling alternative. In the serverless model, users register function images and the frame-work dynamically launches, auto-scales, and terminates function instances on demand, charging only for the resources consumed during execution [ 42 , 54 , 70 ]. This model provides two key bene-fits for fluctuating workloads. First, elasticity : function instances can scale from zero to thousands and back in seconds, matching demand at a granularity far finer than server-level scaling. Second, 

cost efficiency : because the pricing model charges per millisecond of execution, applications with idle intervals between bursts of activ-ity pay only for the time they are active, unlike VMs that incur cost whether busy or idle. These properties have already enabled server-less designs for data analytics [ 67 , 89 ], machine learning [ 39 , 52 ], and video encoding [ 46 ], hinting at the potential of serverless com-puting to host a broader class of backend services. Distributed file systems are natural candidates to benefit from these properties. Most file operations are simple and short-lived— translating file offsets to block addresses, reading or writing a few blocks—and many applications issue file I/Os with intervals that leave the file system idle. Capturing these operations in server-less functions can bring fine-grained elasticity [ 40 ] and, when idle intervals are reflected in the workload, significant cost savings com-pared to always-on server deployments. For example, consider a multi-tenant platform where a media pipeline generates bursts of small-file writes during upload windows then goes idle for hours, a nightly analytics job reads terabytes once per day, and a web application sporadically accesses configuration files. Today, these tenants either share an over-provisioned file system—paying for peak capacity at all times—or each deploys a dedicated cluster, multiplying operational overhead. A serverless file system can in-stead scale each file’s serving capacity independently, from zero to many concurrent function instances, charging only for the I/O actually performed. 𝜆 FS [40 ] explores this direction by extending HopsFS [ 63 ] to delegate file metadata service to serverless func-tions. However, 𝜆 FS keeps function instances alive even during idle periods to cache metadata, forgoing the cost benefits of the serverless model. In this paper, we propose, to our knowledge, the first serverless file system, SLFS , which uses serverless functions to manage both file data and metadata. Serverful storage systems exist for serverless applications [ 51 , 55 , 69 , 88 ], but SLFS uses serverless functions as its main building blocks and can be used by both serverless and non-serverless applications. 𝜆 FS only handles metadata using functions and is intended for large file accesses like its predecessors, HopsFS

and HDFS. In contrast, SLFS is designed from scratch for generic files, employs simple metadata designs suited to the serverless environment, places both file data and metadata operations on serverless functions, and adds mechanisms to efficiently utilize functions at low cost. A key design decision in SLFS is to implement file operations on top of key-value stores, keeping the computation within serverless functions simple and short. Rather than managing complex on-disk structures such as inodes and block bitmaps inside ephemeral func-tions, SLFS uses hash-based mappings from file paths and block numbers to key-value pairs, similar to how Ceph [ 83 ] uses hash-ing to map objects directly to storage nodes without centralized metadata lookups. This design is a natural fit for the serverless model: functions perform straightforward key-value gets and puts, avoiding the heavyweight state and long-running logic that would conflict with short function lifetimes. However, unlike Ceph, SLFS 

does not manage the key-value backend itself; it delegates man-agement complexity such as replication and load balancing to the storage backend. Thus, SLFS can run on any storage system with key-value APIs, taking advantage of existing cloud key-value ser-vices [ 14 , 18 , 22 ] as well as users’ own key-value store implementa-tions [ 15 , 23 , 30 ]. The backend choice can be diversified depending on performance, cost, management, and other requirements. The financial cost of operating SLFS is configurable and signifi-cantly cheaper than using a cloud-native file system or 𝜆 FS . Cloud providers charge higher rates to store and access data in a file system than in a simple key-value store or a block store, but there are still application demands for file services. SLFS targets such applications, and our analysis of Azure traces [ 28 ] under various SLFS configura-tions reveals that the operational cost of SLFS is cheaper than that of cloud-native file systems (Section 2.2). 𝜆 FS reduces cost compared to open-source file systems such as HDFS [ 7] and HopsFS [ 63 ], but SLFS reduces cost further by dynamically controlling function lifetime while maintaining high performance (Sections 8.4 and 8.7). Despite these advantages, realizing a serverless file system is non-trivial and entails unique challenges that stem from the serverless environment. Like any serverless application, SLFS must deal with the cold start problem [ 42 ]. Depending on the serverless platform, starting a function from a cold state (i.e., starting from scratch) can take a few seconds, and even from a warm state (i.e., starting from a paused sandbox with a loaded function image) can take over 100s of milliseconds [ 87 ]. To mitigate cold starts and save function invocation costs, SLFS introduces a novel function instance design as a short-lived multi-threaded server. This design induces hot starts (i.e., processing requests with a fully active function instance) and keeps the instance busy even when waiting for an I/O response. Supporting atomicity and isolation over weakly consistent back-end storage systems is another significant challenge in the server-less environment [ 51 , 73 , 88 ]. SLFS dynamically assigns a dedicated function instance for each file for isolation: that is, a file is encapsu-lated in a function instance and is mutated by calls defined in the instance, following an object-oriented design [ 61 ]. This design also facilitates a file cache within function instances. SLFS leverages atomic commit features of backend key-value stores for consistent updates and uses the two-phase commit protocol when the back-end lacks native atomic commit support. SLFS provides single-file atomicity, and atomic multi-file operations such as rename. Because most serverless frameworks do not fully support in-bound networking of function instances, mapping a file to a func-tion and forwarding the file request to the dedicated function is challenging. SLFS adds a coordinator to control the request flow to the functions. The coordinator enforces SLFS policies, mitigates cold starts, load-balances requests across function instances, and scales the system. We demonstrate through evaluation that SLFS is efficient, flex-ible, scalable, and cost-effective. We compare SLFS against In-finiCache [ 81 ], 𝜆 FS [40 ], AWS Elastic File System (EFS) [ 9 ], and Ceph [ 83 ] and show that SLFS delivers comparable or better per-formance at lower cost. SLFS takes advantage of various storage backends while mitigating cold starts by 580 × compared to default serverless configurations, improving performance significantly, and reducing cost by up to 63%, 68%, and 63% compared to 𝜆 FS , EFS, and Ceph, respectively. 

SLFS proposes a new way of constructing a distributed file system using the serverless paradigm, offering a cost-effective alternative to existing cloud file services [ 9 , 17 , 20 , 83 ]. This paper makes the following contributions: 

• We propose the first serverless file system that uses serverless functions for handling both metadata and data. 

• We present a novel short-lived-server-based function instance design, a coordinator design, and policies for the serverless file system to reduce cold starts, save cost, and scale. These designs are not limited to SLFS and can be applied to other serverless systems and applications. 

• We implement SLFS , deploy it in the public cloud, and evaluate it under various scenarios to demonstrate its strengths and weaknesses against other systems. 

## 2 Background and Motivation 2.1 Cold Start Problem 

Serverless computing allows the convenience of on-demand execu-tion of functions by launching the code and the runtime environ-ment, usually in a container. The trade-off for this approach is that if the function has not been executed for a while, it takes a long time to fetch the image of the container and start the runtime envi-ronment. Mitigating the delays from this cold start problem is an active research area. There are solutions such as creating a lighter-weight sandbox [ 33 ], using snapshots of the container to reduce the start-up time [ 38 , 44 , 79 ], and predicting the task invocation patterns [ 47 , 68 ] to keep the container in a warm state (i.e., loaded but in a dormant state) for quick reuse. Some cloud providers offer a service to keep the function instance warm (e.g., Azure Durable functions and SnapStart [ 2 , 16 ]) at a higher price. SLFS can benefit from these but proposes a more proactive methodology and policy to keep the function hot while maintaining a low cost (Section 5).

## 3 SLFS Design Overview 

In this section, we introduce the high-level design of SLFS . Then, we dive into detailed designs of SLFS and policies to overcome unique challenges in the following sections. The design of SLFS is illustrated in Figure 2. SLFS is composed of the grey boxes in the figure. Main computations to support the file operations are designed as serverless functions: the data function ( 𝑓 𝐷𝑎𝑡𝑎 ) is responsible for accessing data files, whereas the metadata function ( 𝑓 𝑀𝑒𝑡𝑎 ) handles metadata operations. The file data is broken into fixed-size blocks and stored as key-value pairs inside the backend storage services. SLFS is designed to run on any cloud- or user-managed key-value stores. Similar to the Ceph [ 83 ] file system, SLFS uses the hash key of a file path and a block number to map the block of the file to a key-value pair location. 𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 operations are designed to be minimal for low-cost function invocations. The coordinator orchestrates the function accesses. File requests first go through the coordinator. The coordinator initializes the function instance dedicated to the file and establishes direct con-nections between the function and the client. The coordinator is the only serverful component in SLFS , which is similar to a proxy in other serverless system designs [ 40 , 81 , 90 ]. We augment this de-sign to avoid cold start, control concurrency, load-balance requests, and save SLFS costs. The coordinator information is managed by the membership service natively in the serverless framework. The coordinator also scales up and down and the membership service in the serverless framework manages its information. 

## 3.1 Client Libraries and SLFS APIs 

The client library retrieves the coordinator information from the membership service and establishes connections to 𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 

with the help of the coordinator (Section 3.4). 

SLFS and the client library offer relaxed POSIX-compliant APIs similar to existing production and open-source distributed file sys-tems [ 7 , 9, 49 ]. The APIs include most file operations including open, close, read, write, delete, and rename and metadata operations. The open call registers the file to the coordinator, and the co-ordinator assigns and launches 𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 for the file. The close call ensures that any outstanding I/O for the file is completed and arXiv.org, July 2026, Boston, MA, USA Cheng Hao (Ryan) Yang, Paola Alsharabaty, Soufiane Jounaid, Cristina Nita-Rotaru, and Ji-Yong Shin 

deregisters the file unless other users are accessing it. The file-to-function mapping can be dynamically changed by the coordinator, and long-term inactive files need remapping to a new function. Directories are implemented as files containing a list of files. Readdir reads the directory file and makedir creates a new direc-tory file. Files and directories can be renamed, but this entails an overhead of copying the file and directory to new locations because the hash of the file path indexes files. 

SLFS guarantees per-file atomicity for multi-block operations, serialized access for concurrent operations on the same file, and atomic multi-file operations such as rename; we specify these guar-antees precisely in Section 4.3. 

## 3.2 Data and Metadata Functions 

A key design goal for SLFS functions is to keep operations simple and fast so that function invocations remain short and inexpensive. To achieve this, SLFS implements file operations on top of key-value stores using hash-based indexing: read and write requests are translated into block-granularity key-value accesses, where the hash of a file path and block number directly maps to a key-value pair location without traversing any sophisticated metadata structures. This design is critical in the serverless environment. Serverless functions are short-lived and not well-suited for caching inode-like metadata, and accessing a persistent backend store for such metadata over the network can cause huge delays. Indeed, our early prototype design with inodes was unusably slow and we had to switch to the hash-based design. In contrast, 𝜆 FS [ 40 ] caches file metadata in function instances and extends the function lifetime to the maximum to overcome this problem, but it incurs high costs and performs poorly when writing metadata (Section 8.5). Building on this simple operation model, we propose a novel design for the function instance: a short-lived concurrent server that executes simple tasks while keeping the function invocation cost low. Unlike regular serverless functions that terminate after process-ing a single request, our function instances run as multi-threaded servers that accept and process multiple requests concurrently dur-ing their lifetime. The data function ( 𝑓 𝐷𝑎𝑡𝑎 ) handles file read and write requests by performing key-value gets and puts based on the hash key. The metadata function ( 𝑓 𝑀𝑒𝑡𝑎 ) is similar but is dedi-cated to directory accesses; directory files are managed the same as regular files, but 𝑓 𝑀𝑒𝑡𝑎 implements directory-specific operations. Because these operations are lightweight, they are well-suited to be processed concurrently within the server-based function in-stance, keeping the instance busy even when individual threads are waiting for backend I/O responses. The lifetime of each instance is controlled by a policy that balances cost and performance: the instance stays alive long enough to exploit temporal locality and induce hot starts, but terminates when idle to avoid unnecessary cost (Section 5). The 𝑓 𝐷𝑎𝑡𝑎 and 𝑓 𝑀𝑒𝑡𝑎 encapsulate file and directory instances as in object-oriented (OO) style designs. Each function instance knows how to mutate and access a file or directory, and SLFS assigns at most one dedicated function instance per file or directory. Thus, accessing the function instance becomes analogous to accessing a file/directory object. The OO design cleanly isolates files/directories and facilitates concurrency control (Section 4.3). ## 3.3 Coordinators 

The coordinator maps a file to a function instance. This mapping policy determines the performance and cost of SLFS and keeps concurrent file accesses safe (Section 5). Upon opening a file, the coordinator checks if the file is already open. If not, the coordinator maps the file to an existing function or starts a new one to map the file. The mapping information is kept until no user accesses the file or the function terminates. Between the open and close of a file, different functions can service the file requests, but the coordinator ensures that at most one function instance serves the file at any given moment. The coordinator can scale up or down depending on the load. 

SLFS uses consistent hashing like Chord [ 53 , 75 ] to assign files to coordinators. However, unlike Chord, all coordinator information is kept in a centralized membership service in the serverless frame-work for consistent member updates and one-hop request delivery to the coordinator.

For SLFS operations, the coordinator tells the client about the function instance in charge of the file that the client is trying to access ( 1 and 2 ). If a new function instance is needed to process the client request, the coordinator starts a new function instance ( 2 ) before returning to the client. ## 3.5 Storage Backend 

SLFS is designed to work with key-value storage backends. The condition for safety (see Section 4.3) is that the key-value store should provide read-my-writes and bounded staleness, or stronger guarantees [ 77 ]: users must always see their own updates, and an update must become visible to everyone after a bounded time. Most cloud storage systems, such as S3 [ 18 ], fulfill this requirement. To further demonstrate a more general use case of SLFS , we also consider even more primitive storage building blocks based on standalone VM-based key-value storage servers (Section 7). 

## 4 Detailed Designs 

We describe the detailed SLFS designs to answer the following questions for realizing a practical serverless file system: 1) how to avoid cold starts; 2) how to lower the financial cost; 3) how to scale the components; 4) how to maintain consistency and 5) how to handle failures. Figure 4 illustrates the detailed relationships and roles of SLFS components. 

## 4.1 Avoiding Cold and Even Warm Starts 

Cold-starting a function instance can take a few seconds, and even warm-starting an instance can take 100s of milliseconds [ 87 ]. The recent AWS SnapStart feature reduces the start-up time to a few milliseconds [ 2] with a warm snapshot of a function instance, but for fast file accesses, it is best to even avoid the warm start. We propose a novel short-lived multi-threaded server-based function design for SLFS to induce a hot start. 

Function Instances as Short-Lived Servers. We design 

𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 as short-lived servers to reduce the file access latency. Unlike regular serverless function instances that terminate right after processing a request, SLFS functions wait for new incoming requests for a short period leveraging the temporal locality of file accesses. If there is another request to the same file within a short interval, the instance can be immediately reused; this leads to a hot start. The limitations are that cold or warm starts are unavoidable under infrequent I/Os, and extra costs could incur for idling. Thus, the function lifetime policy is designed to keep the cost low while inducing hot starts (Section 5.2). 

Multiple Files Per Function Instance. If a function instance is only mapped to a single file, the frequency of the instance reuse will be low, and new file accesses will entail cold starts. To better utilize a function, we assign multiple files per function (Section 5.1) and make the instance multi-threaded. For security reasons, however, only the files of the same application/user share a function. Concurrently handling different file requests in the same func-tion instance saves function invocation costs: e.g., cold-starting one instance prepares multiple file requests to be processed immedi-ately. Without concurrent request processing, functions could idle while waiting for I/O responses from the backend storage, but our design keeps the function busy and reduces the overall function execution time. As long as the user frequently accesses any file the utilization of the function instance can increase, and cold and warm starts can be avoided. Thus, for example, scanning through a list of files or recursively reading file lists in directories (e.g., find or ls -R) is handled efficiently using only a few functions. 

Provisioning Replacement Functions. Even though the 

𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 can handle multiple requests for different files with-out having to terminate after processing a request, it has a limited lifetime set by the serverless framework [ 71 , 82 ]. In case there is a need to process I/O requests beyond the function’s lifespan, each instance keeps track of its duration. The function instance notifies the coordinator when it is close to the termination time set by the framework. Then the coordinator launches a replacement function early enough to hide the cold start. Later, the old function termi-nates with notifications to the coordinator and connected clients, and the coordinator directs requests to the new function. 

## 4.2 Scaling Up and Down 

For any serverless application, scaling up and down is important for both performance and cost. The scalability of SLFS is only bounded by the backend storage that SLFS relies on. 

Scaling Coordinators. The coordinator scales up and down depending on the user load. When the number of open files a coordi-nator manages increases, the coordinator spawns a new coordinator server to partition the range of files it manages. The existing coor-dinator reorganizes the file to function mapping, function instances that manage files for the new coordinator connect to the new coordi-nator, and the new coordinator updates its file to function mapping. The new coordinator becomes active after its information is reg-istered with the membership service. Due to the consensus-based membership service (i.e., Zookeeper [ 4 ]) the function instance map-ping and the hand-off of a partition take place consistently and then clients of SLFS are notified of the changes through the client library. When the load on the coordinators goes below a watermark, 

SLFS merges adjacent coordinator nodes through the membership service similar to the scaling up procedure. Overall, scaling up a coordinator is much more costly than scaling a function instance, so 

SLFS triggers the merger when the load is kept below the watermark for an extended period. 

Scaling Data and Metadata Functions. Because we map mul-tiple files to a function instance, SLFS should opportunely launch new instances as the load increases. SLFS monitors the request queue length and starts new functions if the queue length exceeds a threshold. It considers the cold start latency and proactively provi-sions function instances for the best performance (Section 5.3). ## 4.3 Maintaining Consistency 

Guarantees. Before describing the mechanisms, we summarize the guarantees SLFS provides to clients: (1) reads and writes to a file spanning multiple blocks are atomic; (2) concurrent accesses to the same file are serialized; (3) multi-file operations such as rename are atomic; and (4) completed writes are durable. 

File Consistency. SLFS uses a mix of soft-update [ 48 , 62 ] and lock-based two-phase commit [ 34 , 37 ] to guarantee file consistency. 

SLFS does not employ an explicit index such as the inode, and lo-cations of file blocks are derivable by a hash function. The only persistent index-like metadata is the directory file, and files synchro-nize through the directory. For example, the directory is updated after a file is created, and the file is deleted after the directory entry of the file is removed. If the operation fails halfway, there could be orphaned files, which can later be garbage collected, but inconsistent states are not revealed to the user. Thus, the directory information acts as the ground truth for what is in the file system. Note that concurrent updates to the directory file are safe because a dedicated 𝑓 𝑀𝑒𝑡𝑎 serializes the concurrent operations for each di-rectory file. For multi-directory/file updates (e.g., rename), one of the functions registers the transaction with the coordinator and works as a transaction coordinator to drive the two-phase commit protocol [37] over the functions that own the relevant files. For atomic reads and writes to multiple blocks of the same file, we leverage the transactions or atomicity features [ 1, 10 ] supported by the backend key-value stores. In case no such features are sup-ported, SLFS uses the two-phase commit to atomically read or write multiple blocks. This feature is activated when SLFS runs on top of our custom backend key-value stores (Section 7). 

Replicated Consistency. Depending on the backend key-value store and its replicated consistency semantics, stale versions of a file can be exposed. However, as long as the backend key-value store supports read-my-writes or stronger semantics [ 76 , 77 ], SLFS 

can service the up-to-date state of all files it manages and preserve consistency across different key-value pairs. Note that the weak consistency guarantees such as read-my-writes are often defined per user basis but the user of the key-value store in our context is the entire SLFS which directly interacts with the key-value store. Such consistency guarantees for the entire SLFS extend strongly to individual end users. For example, under read-my-writes, an end user’s completed write becomes immediately visible to another end user even though they are different users, because SLFS as a single user of the key-value store reads and writes on behalf of all end users. All backend stores used for SLFS in this paper, including Amazon S3 and our own key-value store implementation, satisfy this requirement. 

## 4.4 Handling Failures 

Similar to most serverless applications, SLFS functions are stateless, and critical states are maintained in the persistent backend store. We assume the backend store is durable (e.g., due to triple replication) and the membership service is highly available (e.g., backed by Paxos-like protocols [ 4 , 59 ]). With these assumptions, SLFS can tolerate the failure of data and metadata functions and coordinators and maintain consistency. 

Failures of Data and Metadata Functions. 𝑓 𝐷𝑎𝑡𝑎 and 𝑓 𝑀𝑒𝑡𝑎 

are stateless worker functions managed by the coordinator. If they fail, the coordinator can detect the failure, reassign the files of the failed function to other functions, and redirect the file requests ac-cordingly. If the coordinator is unsure whether a function instance has failed or not, it can ask the serverless framework to terminate the suspected instance and switch to a new one. Any queued re-quests in the failed function will be returned as failed I/O or timeout, and users are responsible for re-executing them. SLFS guarantees the atomicity of file I/O requests, so function failures do not cause inconsistencies. Still, SLFS needs to abort or complete incomplete I/O operations involving two-phase commits which follows the two-phase commit recovery protocol of checking transaction intention records left behind in the backend key-value stores. 

Failures of Coordinators. The coordinator manages states that outlive the 𝑓 𝐷𝑎𝑡𝑎 and 𝑓 𝑀𝑒𝑡𝑎 , and handling its failure is more complicated than that of function instances. SLFS uses the heart-beat mechanism to detect failures using the membership service where the coordinators are registered. If the failure is detected, a replacement coordinator is registered, and an epoch number is incremented. Functions of the failed coordinator can continue pro-cessing requests but they terminate when they cannot communicate with the coordinator; the new coordinator starts operating after all old functions terminate. 

Recovery of In-Flight Transactions. Multi-file transactions in SLFS (e.g., rename) use two-phase commit, where one function instance acts as the transaction driver. To enable recovery under failures, the driver writes a transaction intention record containing the participant list, transaction ID, and current state to a well-known key prefix in the backend key-value store before sending prepare messages to participants. Each participant records its vote (commit or abort) in the backend store as well. This design enables recovery even under simultaneous failure of the coordinator and the driving function. When a new coordinator starts after a failure, it scans the backend store for pending intention records. For each pending transaction, the new coordinator deter-mines the outcome: if all participants have voted to commit and a commit record exists, the transaction is completed by ensuring all participants apply the update; otherwise, the transaction is aborted and participants roll back. This follows the standard two-phase commit recovery protocol [ 37 ]. The overhead of this mechanism is one additional key-value write (the intention record) per multi-file transaction. In our workloads, multi-file transactions are rare, so the overhead is negligible. 

## 5 SLFS Policies 

SLFS maintains internal policies to make decisions for 1) how to map a file to a function instance, 2) how long the function instance should wait for new requests, and 3) when to launch new function instances. The policies affect and consider the performance and the cost of running SLFS .SLFS : a Flexible, Low-Cost Distributed File System Using Serverless Designs arXiv.org, July 2026, Boston, MA, USA 

## 5.1 File-to-Function Mapping 

SLFS file requests go through the coordinator, so the coordinator keeps track of how many files are opened and how many I/O re-quests for each file are queued in the function. We have explored random and greedy policies to map a file to a function. The greedy policy assigns files to the function with the shortest request queue size. Based on an analysis using the Azure serverless I/O trace, we find that the greedy policy is simple yet works the best in general and use it for SLFS . Even after the initial assignment, the file-to-function mapping can change dynamically. The coordinator periodically monitors the load on each file and remaps the heavily accessed files to the least loaded or a new function instance. 

## 5.2 Function Lifetime 

An extended function lifetime trades off function running cost for low latency. Keeping the function hot all the time as in 𝜆 FS [40 ]yields the best performance, but this nullifies the purpose of the demand-based serverless design. While other systems mathemati-cally predict the load to keep the function instance warm [47 , 68 , 71 ], 

SLFS maintains the functions hot for a shorter period with higher sensitivity. Our policy is, therefore, simpler and uses a moving average of request arrival intervals of the last 𝑁 requests for each function. The function instance waits for 𝑇 𝑒𝑥𝑡 , the average interval with an added error margin 𝜖 , after it processes all queued requests. However, if the request arrival intervals are very short, reflecting close-to-zero intervals to 𝑇 𝑒𝑥𝑡 does not make sense. Thus, we only consider intervals greater than a threshold 𝑇 𝑚𝑖𝑛 . To reduce the effect of outliers, we provide a safety threshold using the coefficient of variation (standard deviation / mean). If the coefficient is too high, we revert to a default 𝑇 𝑑𝑒 𝑓 𝑎𝑢𝑙𝑡 until the spread gets smaller. 

## 5.3 Launching New Functions 

Because the serverless pricing model charges for the duration of a function, the price for serially invoking the same number of func-tions is the same as concurrently invoking them. SLFS functions internally parallelize and pipeline tasks and extend their own life-time, so opportunely launching new functions is critical to improve performance and save costs. 

SLFS starts a new function instance 1) when a function instance is about to be terminated by the serverless framework due to the maximum run time limit, and 2) when there are not enough func-tion instances to handle the load. In the former case, the SLFS 

coordinator starts a replacement function while the terminating function is running to switch and hide the cold start latency. For the latter case, SLFS profiles the steady-state performance of 𝑓 𝐷𝑎𝑡𝑎 

and 𝑓 𝑀𝑒𝑡𝑎 and measures the maximum load that the functions can handle without sacrificing performance. The coordinator monitors the load based on the function’s feedback and launches new func-tions. If a burst of requests arrives, the coordinator starts multiple functions proportional to the load in parallel. In this case, cold or warm starts are unavoidable. However, the multi-threaded design of the function handles the load more efficiently with a smaller number of instances than regular serverless applications that start an instance per request. ## 6 Caching 

We implement a write-through least recently used (LRU) block cache inside each function to increase the performance and mem-ory utilization (Figure 5). SLFS places the cache inside the function instance for three reasons. 1) SLFS function instances usually un-derutilize allocated memory, so the cache comes almost for free. 2) 

SLFS files are mapped to dedicated function instances, so there is little cache contention, and 3) a single copy of cached data can serve all requests to a specific file without cache consistency concerns. The write-through approach by the dedicated function retains dura-bility and consistency. Two levels of hashing using file names and block numbers index the cached data. A unique challenge in the serverless environment is that a func-tion instance with cached data frequently terminates. Before the in-stance terminates, it sends the information about the cached blocks (not the data) of each file (2nd-level hash table) to the coordinator. When the coordinator remaps the file to a new function instance, this cache information is sent to the new instance. Then, the new instance prefetches recently accessed 𝐾 blocks of the file from the backend key-value store into its cache for continued caching. The cached block information that is temporarily stored in the coordi-nator is small. However, the coordinator keeps a cap and evicts the block information at file granularity based on the LRU policy. Note that the typical cache information size per function is only a few hundred bytes, and the cache can operate without the information transfer.

In this section, we evaluate the basic performance and scalabil-ity of SLFS , how well SLFS avoids cold and warm starts, and the performance and cost of SLFS against other systems. 

Implementation and Basic Configurations. We implement 

SLFS on OpenWhisk [ 3] running with Docker containers [ 19 ]. We use the Zookeeper included in the OpenWhisk package as the SLFS 

membership service. All of the SLFS components, including the user library, coordinator, 𝑓 𝐷𝑎𝑡𝑎 , 𝑓 𝑀𝑒𝑡𝑎 , and PSB, are implemented in 12.6K lines of C++ code. 𝑓 𝐷𝑎𝑡𝑎 and 𝑓 𝑀𝑒𝑡𝑎 are integrated as a single Docker image for easy reuse of the function instance. We use 7 VMs to host components related to the serverless framework, which include the OpenWhisk controller, Zookeeper, and coordinators, 16 VMs to run OpenWhisk workers and Docker-based function instances, and 9 VMs to set up the backend storage. Therefore, experiments in Sections 8.5 and 8.6, which feed in a large continuous stream of I/O workloads, demonstrate the peak performance of SLFS , but cannot represent a scenario for cost comparisons of running the system in production.

## 9 Related Work 

Serverless In-Memory and Persistent Storage Systems. Infini-Cache [ 81 ] builds a distributed in-memory cache out of serverless function instances. SLFS uses the coordinator and InfiniCache uses the proxy to relay the request to function instances. However, SLFS 

further extends the role of the coordinator to make policy-based de-cisions for hot starts and better resource management. InfiniCache does not offer data durability and heavily relies on warm function containers over which InfiniCache has little control. InfiniStore [ 90 ]extends InfiniCache with a serverless memory abstraction and per-sistence using a cloud object store to tolerate failures. However, it does not support file abstractions and lacks the fine-grained func-tion control of SLFS .

𝜆 FS augments HDFS-based HopsFS [ 63 ] with the InfiniCache idea to cache only metadata in a long-lived function instance. Its coordination service and name nodes resemble the coordinator and 

𝑓 𝐷𝑎𝑡𝑎 /𝑓 𝑀𝑒𝑡𝑎 of SLFS but SLFS is more general and supports fea-tures not in 𝜆 FS . SLFS handles both file data and metadata through finely managed functions, dynamically controls function lifetime for performance and cost, and is portable to different backends. 

Cold Start and Function Lifetime Management. Cold start problems are largely tackled by exploring lightweight mechanisms to start the function instance [ 33 , 38 , 44 , 79 ] and keeping function instances warm based on predicted request patterns [ 47 , 68 , 71 ]. The SLFS solution is closer to the latter but extends the hot function state for a short controlled period to avoid both cold and warm starts; this is complementary to existing approaches and opens up new opportunities to even avoid warm starts. 

Support for Stateful Serverless Computing. Pocket [ 55 ] is an ephemeral storage system for short-lived serverless tasks, Boki [ 51 ]realizes a distributed shared log for serverless functions to coordi-nate, and Cloudburst [ 74 ] extends the auto-scaling key-value store Anna [ 85 ] with HydroCache [ 86 ] to collocate data and computa-tion with a causal consistency guarantee. Similar to these systems, 

SLFS improves the accessibility of state across different serverless functions over persistent backend storage systems. Beldi [ 88 ] imple-ments a serverless transactional key-value store with exactly-once semantics. Like Beldi, SLFS leverages the atomic commit features provided by the backend storage system and distributed transac-tions for consistent updates. SLFS does not guarantee exactly-once semantics, which is beyond what most file systems provide, but Beldi’s approach applies complementarily. Among all these systems, 

SLFS is the only file service leveraging serverless functions. 

Decomposed File Systems. There are serverful distributed [ 83 ,84 ] and local [ 50 , 56 , 64 , 72 ] file systems designed on top of key-value stores. While local file systems leverage key-value stores for more efficient indexing and fast metadata access, distributed file systems use them for modularity of design and ease of system management. FUSE-based file systems on cloud backend key-value stores [ 21 , 31 , 60 ] are designed to leverage low-cost cloud key-value stores. While SLFS also uses key-value stores to delegate the complexity of fine-grained data management and to utilize low-cost cloud backend stores, SLFS goes further to achieve utmost elasticity and even lower costs through serverless designs. Existing systems may be able to leverage container as a service at best without code modification for elasticity, but due to a heavier code footprint and larger image size, their startup time would be nearly a minute [ 12 ]to tens of seconds [5, 11], so they cannot be as elastic as SLFS .CNFS [ 35 ], a cloud-native file system, advocates for designing storage systems around cloud primitives. CNFS is a local, hierar-chical, copy-on-write file system that migrates data and metadata across cloud storage volumes and offloads background tasks such as migration and compression to remote CPU workers. While CNFS shares the high-level vision of decomposing file system function-ality across cloud services, it focuses on local file system design principles and does not leverage the serverless computing model. 

SLFS is a fully distributed file system that uses serverless func-tions and proposes designs and methodologies to fully leverage the serverless environment. 

## 10 Conclusions 

We presented SLFS , the first distributed file system mainly designed with the serverless paradigm. SLFS can run on cloud-native and users’ custom key-value stores and adjust to users’ needs. SLFS 

implements file operations using serverless functions. The function instances are designed as novel short-lived multi-threaded servers that dramatically avoid cold and warm starts at a low cost. SLFS 

makes policy-based decisions to scale the system elastically and uses residual function instance memory for caching. Our evaluation in the cloud with Azure workloads shows that SLFS performs better than 𝜆 FS , EFS, and Ceph at a significantly lower cost.
