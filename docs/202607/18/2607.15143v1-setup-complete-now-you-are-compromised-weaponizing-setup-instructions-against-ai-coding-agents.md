## 摘要
AI编程代理通过阅读文档并安装其中列出的依赖项来设置项目，而无需验证其名称、来源或已知漏洞。只需修改README、requirements文件或Makefile，攻击者就能将代理重定向到不受信任的注册表、已知存在漏洞的版本或名称相似但错误的包：文档成为代码执行的向量。我们首次系统评估了通过普通项目设置文档在生产级编程代理框架中传递的包安装时供应链攻击，基于已记录的安全事件，针对五类攻击场景中的十二种情况探测前沿模型。同一模型在一个框架中检测到攻击，而在另一个框架中却安装了它：安装时的安全性依赖于框架-模型组合，而非模型本身。代理能够可靠地捕获明显的拼写错误，但看似合理的分隔符混淆名称（如azurecore代替azure-core）却可以蒙混过关，其成功率取决于框架-模型配对。基于来源的攻击（如注册表重定向）几乎在所有地方都被忽略。这一来源盲点在npm和Cargo上反复出现，几乎所有模型都会安装不受信任的依赖；名称检测在不同生态系统间的一致性较差。面向安全的提示可以部分弥补这一差距，但仅限于它们指定的维度；一种确定性安装前检查机制，在任何代码运行前验证名称、来源和版本，可以弥补大部分漏洞。

## Abstract
AI coding agents set up projects by reading documentation and installing the dependencies it lists, without verifying their names, sources, or known vulnerabilities. By editing only a README, requirements file, or Makefile, an attacker can redirect the agent to an untrusted registry, a known-vulnerable version, or a wrong-but-plausible name: documentation becomes a vector for code execution. We present the first systematic evaluation of package-install-time supply-chain attacks delivered through ordinary project-setup documentation across production coding-agent harnesses, probing frontier models on twelve scenarios in five attack classes, grounded in documented incidents. The same model catches an attack through one harness and installs it through another: install-time security rests on the harness-model combination, not the model alone. Agents catch blatant typosquats reliably, but plausible separator-confusion names (azurecore for azure-core) slip through, and how often depends on the harness-model pairing. Source-based attacks like registry redirection are missed almost everywhere. The source blind spot recurs on npm and Cargo, where nearly every model installs the untrusted dependency; name detection carries over less consistently across ecosystems. Security-oriented prompts recover part of the gap but only for the dimension they name; a deterministic pre-install check that verifies names, sources, and versions before any code runs closes most of it.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：当前AI编码代理（如GitHub Copilot、Claude Code、Cursor等）在设置项目时，会自动读取README、requirements.txt、Makefile等文档，并执行其中的安装指令（如`pip install`），而**不验证包名、来源或已知漏洞**。这暴露出一个“安装缺口”（install gap）：包管理器本身不将安装视为安全操作，而AI代理移除了人类在点击前短暂的复查环节，使得文档成为代码执行的攻击向量。
- **整体含义**：攻击者只需篡改项目文档（如提交一个修改README的PR），即可诱导代理安装恶意包（typosquat、依赖混淆、注册表重定向、漏洞版本锁定等），从而在开发者机器上实现代码执行。该攻击面已在真实事件中出现（如torchtriton、Ultralytics、LiteLLM等）。论文系统评估了这一威胁，并指出安全性取决于“代理框架-模型”组合，而非仅模型能力。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：攻击者通过**仅修改项目文档**（README、pyproject.toml、requirements.txt、Makefile）来注入恶意安装指令，AI代理盲目执行，绕过人类审查。
- **关键技术细节**：
  - **威胁模型**：假设攻击者可以（1）在PyPI或自有服务器上注册恶意包（typosquat、名称混淆）；（2）向GitHub仓库提交文档修改（PR或新项目）；（3）运行自己的包服务器并通过`--extra-index-url`引用。攻击者无需入侵PyPI或开发者机器。
  - **安全属性缺失**：安装时必须保证三个属性——AUTHENTIC（来源可信任）、INTENDED（名称正确）、SAFE（版本无已知漏洞），但当前所有代理流程均不验证。
  - **五类攻击向量**：
    1. **名称攻击**：明显typosquat（`tranformers`）、分隔符混淆（`azurecore` vs `azure-core`）、微小转置（`aiohtpt`）、社会工程（`langchain`拼写错误）、跨文件冲突、批量typosquat。
    2. **来源攻击**：依赖混淆（`--extra-index-url`）、不受信任本地/外部注册表、requirements.txt中隐藏索引。
    3. **版本攻击**：锁定已知漏洞版本（CVE pinned）。
    4. **配置投毒**：通过Makefile设置`PIP_CONFIG_FILE`重定向解析。
    5. **输出注入**：通过ImportError错误消息诱导安装恶意包。
- **算法流程**（文字说明）：每个场景构建一个“正常”Python项目，但文档中嵌入特定攻击。代理收到通用设置提示（无安全提示），执行安装。通过`uv pip show`检查最终环境中的包名、来源、版本，与转录中的警告结合，判定“捕获”（caught, 即拒绝安装或修正前拒绝了）或“失败”（failed, 即安装成功或安装后标记）。对于名称类，每个配置跑30次（10个包×3次）；其他10次。

## 6. 论文的主要结论与发现

1. **检测率取决于框架-模型配对，而非仅模型**：固定模型（Opus 4.8）更换框架（Claude Code vs Copilot CLI），对localhost注册表攻击的检测从10/10降至9/30；方向可翻转（外部HTTPS注册表：Claude Code 0/10 vs Copilot CLI 10/10）。
2. **名称攻击基本被解决，但分隔符混淆残留**：明显typosquat和转置几乎100%被检测，但`azurecore`（代替`azure-core`）在部分配置中漏过，比例随框架-模型变化，且不与模型能力等级对齐。
3. **来源攻击几乎全部失败**：所有模型（Opus除外）几乎无条件安装来自不受信任注册表的包。依赖混淆（R5）仅Claude Code和Copilot CLI检测，Codex和Cursor失败。隐藏索引（R8）仅Opus通过Claude Code检测。来源盲点反复出现。
4. **版本攻击完全失败**：R9所有9个配置安装已知CVE版本（0/30），即便Opus事后标记CVE，但安装已执行，属于"install-then-flag"而非防御。
5. **输出注入被全面拒绝**：R7（通过ImportError诱导安装）所有配置100%拒绝，表明代理对来自程序输出的安装指令保持警惕。
6. **安全提示只能针对特定维度缓解，无法覆盖全部**：源定向提示提高隐藏索引检测率，但对版本有效；版本定向提示提高版本检测率，但通用提示无效。
7. **确定性预安装钩子可弥补大部分缺口**：基于名称、来源、版本检查的PreToolUse门控，在测试的11个场景中捕获10个，仅遗漏输出注入类（R7），但其与模型推理互补。
8. **跨生态系统（npm、Cargo）结果与Python一致**：名称检测效果相似，来源攻击几乎全失败，版本攻击只有npm的`npm audit`强制后提醒，但仍是install-then-flag。

## 7. 优点：方法或实验设计上的亮点

- **系统性**：首次对AI编码代理的安装时供应链攻击进行系统评估，覆盖5类攻击、12个场景、多个前沿模型和框架。
- **控制消融实验**：通过固定模型仅换框架，严格分离模型与框架的作用，证实框架本身是决定性因素。
- **确定性评分**：基于`uv pip show`和命令追踪，避免模型自评偏差；使用Wilson置信区间和Fisher精确检验。
- **多包名扫查**：名称类使用40个不同包，避免单点偶然性。
- **跨语言验证**：将相同攻击向量迁移到npm和Cargo，证明攻击是生态系统通用特性。
- **防御验证**：不仅分析问题，还提出了可部署的预安装钩子（开源），并测试了安全提示的局限性，验证了模型内有安全知识但未被激活。
- **实际威胁背景**：引用真实事件（torchtriton、Ultralytics、LiteLLM等），并给出GitHub代码搜索中的流行度估计（typosquat次数、extra-index-url出现次数等），增强研究现实相关性。

## 8. 不足与局限

- **无人类基线实验**：论文未设置人类开发者执行相同任务时的误率比较，因此声称“AI去掉人类防护”是机制性而非对比性。
- **场景实例有限**：每个攻击类只用一个代表性示例（如`tranformers`），虽然名称类有多个包，但来源、版本等类未变化不同域名、不同CVE等，可能影响泛化性。
- **假阳性率检测窄**：仅测试了几个良性项目，未全面评估预安装钩子在大型项目上的误报率（尽管钩子扫描前1000个PyPI包发现了0.5%的误报）。
- **预安装钩子范围受限**：仅拦截`pip install`命令行，未覆盖`uv sync`、`uv run`、PEP 517源码构建等方式；当前仅防御命令字符串，不解析最终解析结果。此外，对自适应攻击者的鲁棒性未测试（攻击者可修改钩子读取的文档）。
- **跨生态系统实验较粗略**：npm和Cargo只使用少数模型和场景，且typosquat注册情况不同（Cargo中`anywow`未注册导致机械性失败），不完全等同。
- **模型版本动态性**：实验在2026年6月进行，模型版本会随时间变化（特别是Cursor router），结论可能随时间漂移。
- **未考虑后门行为**：未测试安装后代理的行为（如持续恶意活动或潜伏）。
