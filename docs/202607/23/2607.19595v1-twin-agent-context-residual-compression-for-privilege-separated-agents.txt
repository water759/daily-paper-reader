## ABSTRACT 

Large language model (LLM) agents are vulnerable to security risks, such as prompt injection attacks from untrusted context that manipulate downstream reasoning and tool use. Existing secure-by-design approaches mitigate this risk by separating untrusted observations from privileged execution and careful control of information flow, but often degrade utility and require extensive task-specific engineering. We thus propose Twin Agent, a general privilege separation design pattern inspired by residual coding in the agent context. Twin Agent consists of two nearly symmetric agents: an Explore Agent that inspects untrusted information and a Safe Agent that executes privileged actions. The Explore Agent is conditioned on the Safe Agent’s current context and communicates only compact hints to the Safe Agent about the next action to take. This design reduces the information needed to preserve task utility and thus achieves a better security–utility tradeoff, which we empirically verify by measuring how utility and attack success change as the length of hints varies. We evaluate Twin Agent on long-horizon software engineering tasks with SWE-bench Lite and on heterogeneous multi-tool interaction tasks with AgentDojo and DecodingTrust-Agent. Across both benchmarks, Twin Agent preserves high task utility while preventing prompt injection attacks, outperforming both undefended agents and privilege separation baselines. 

## 1 INTRODUCTION 

Large language model (LLM) agents increasingly operate in open environments where they must process untrusted information while retaining the ability to perform privileged actions, such as code execution, file-system access, web browsing, or API calls (Yao et al., 2023; Anthropic, 2024; OpenAI, 2025; Anthropic, 2025). This creates a fundamental security challenge for agentic systems: untrusted inputs can manipulate downstream reasoning and tool use through prompt injection attacks, leading to harmful or unintended actions (Willison, 2022; Perez & Ribeiro, 2022; Greshake et al., 2023; Liu et al., 2024). 

Secure-by-design strategies address this challenge by separating untrusted observations from privi-leged execution. A typical Dual LLM design pattern (Willison, 2023) leverages privilege separation to assign different trust levels to different components, allowing one component to observe untrusted information while another retains authority over privileged actions (Provos et al., 2003; Debenedetti et al., 2025; Beurer-Kellner et al., 2025). By explicitly restricting information flow across trust boundaries, these approaches provide a principled path toward secure agent design. Existing secure-by-design methods, such as CaMeL (Debenedetti et al., 2025), often adopt plan-first patterns that perform privileged planning before interacting with untrusted environments. However, many realistic agentic tasks require iterative exploration, environment feedback, and dynamically changing agent trajectories (Yao et al., 2023; Yang et al., 2024; Anthropic, 2024). In such settings, the plan-first design pattern can struggle to adapt to long-horizon interactive tasks. More recent techniques relax this rigidity by allowing restricted communication, such as boolean and enum values, to flow to the privileged agent (Costa et al., 2025; Jacob et al., 2026). However, the untrusted content may contain large amounts of mixed relevant and irrelevant information, making it difficult to design a fixed communication protocol that preserves utility while preventing attacks. As a result, these systems often suffer from severe utility degradation and require manually designed workflows, rigid communication interfaces, or task-specific security policies, making them difficult to adapt across heterogeneous agentic tasks. We argue that secure agent design is better understood as a compressed information flow problem for communication among agents. Attacks such as prompt injection require the transmission of sufficient attack-carrying information to override the intended behavior of the privileged agent. Empirically, we observe that attacking the latest, strongest models requires long injections, so the attacker needs to send a lot of information to succeed. In contrast, preserving task utility often requires only the information necessary to support the agent’s next useful action. As illustrated in the left panel in Figure 1, a robust yet helpful defense should therefore maximize the separation between two quantities: the amount of information needed to preserve utility-oriented actions and the amount of information needed to induce behavioral override. This perspective unifies several existing defense paradigms. Guardrails, monitors, and model-based defenses increase the amount of information required for a successful override by reinforcing desired behavior or filtering suspicious instructions (Chen et al., 2024; Piet et al., 2024; Chen et al., 2025a; Jacob et al., 2025; Li et al., 2025; Chen et al., 2025b). In contrast, privilege separation architectures reduce the amount of information allowed to cross the trust boundary (Willison, 2023; Debenedetti et al., 2025; Beurer-Kellner et al., 2025; Costa et al., 2025; Jacob et al., 2026). Under this principle, and inspired by residual coding (Lu et al., 2019), we propose Twin Agent , a privilege-separated agent design that aims to preserve utility while controlling information flow from untrusted sources. Twin Agent consists of two nearly symmetric agents: an Explore Agent and a Safe Agent. The Explore Agent can observe untrusted information, but cannot execute privileged actions. The Safe Agent can execute privileged tools, but cannot directly observe untrusted information. Instead of passing unconstrained input to the Safe Agent, Twin Agent uses a restricted hint channel: conditioned on the Safe Agent’s limited task state, the Explore Agent communicates only the task-relevant residual information needed for the Safe Agent’s next decision. Figure 2 compares Twin Agent with alternative agent designs. The key idea is that secure communication is more effective when it is differential rather than absolute. Because the Explore Agent is conditioned on the Safe Agent’s current context state, most of the information is already shared. At each step, the Safe Agent does not need a complete sanitized summary of the untrusted context; it only needs the minimum context required to decide the next privileged action. Twin Agent therefore communicates compact residual information rather than retransmitting complete observations or another agent’s context. This design has two benefits. First, it improves the security–utility tradeoff, making it controllable and measurable. Rather than redesigning a new secure workflow for each different domain, Twin Agent is a universal defense that preserves the original agentic interaction pattern and inserts a lightweight residual communication layer between untrusted observations and privileged execution. This facilitates the secure adaptation of existing utility-oriented agents with small architectural modifications, and it makes Twin Agent particularly suitable for long-horizon and adaptive tasks where pre-specified plans or rigid-typed interfaces are difficult to maintain (Debenedetti et al., 2025; Costa et al., 2025; Jacob et al., 2026). Third, Twin Agent partially disentangles utility preservation from security enforcement. Under a fixed communication budget, utility is primarily determined by the Explore Agent’s ability to extract task-relevant information, while security is primarily determined by the isolation and security controls of the Safe Agent. We evaluate the Twin Agent design on two complementary classes of agentic systems. First, we evaluate on SWE-bench, representing long-horizon software engineering tasks that require adaptive exploration and iterative debugging (Jimenez et al., 2024; Yang et al., 2024). Second, we implement tool-using agents for AgentDojo (Debenedetti et al., 2024) and DecodingTrust-Agent (Chen et al., 2026), which represent heterogeneous multi-tool interaction environments with dynamically changing tool outputs and interaction states. Across all benchmarks, Twin Agent shows high utility and resistance to prompt injection attacks. Our contributions are summarized as follows: • We propose Twin Agent , an agentic defense based on privilege separation that preserves utility, features strong robustness against prompt injection, and requires limited task-specific redesign. • We formulate secure agent design as a compressed information flow problem. This perspec-tive motivates a principled relaxation of strict privilege separation, where the security–utility tradeoff can be controlled by limiting the amount of information allowed to cross the trust boundary. • We evaluate Twin Agent on both long-horizon adaptive reasoning tasks and heterogeneous multi-tool interaction benchmarks. Experiments show that Twin Agent achieves high task utility while preventing prompt injection attacks, outperforming both undefended agents and strongly isolated privilege separation baselines. 32 RELATED WORK 

Prompt injections attacks. Prompt injection attacks embed malicious instructions directly in user inputs or indirectly in external content, causing LLMs to confuse untrusted data with trusted instructions and follow attacker-specified objectives (Perez & Ribeiro, 2022; Greshake et al., 2023; Liu et al., 2024). For LLM agents, such attacks can manipulate multi-step reasoning and induce unauthorized or harmful tool actions (Debenedetti et al., 2024; Zhan et al., 2024). 

Model-Based defenses. Many existing prompt injection defenses focus on improving the robustness of the underlying model itself. Detector-based approaches train classifiers to identify malicious inputs and act as filters before the downstream agent or model (ProtectAI.com, 2023; AI, 2024; Wan et al., 2024; Jacob et al., 2025; Li et al., 2025; Meta, 2025). Fine-tuning approaches directly train foundation models on prompt injection attacks to improve robustness against adversarial instructions (Chen et al., 2024; Piet et al., 2024; Chen et al., 2025a;b). These methods can improve robustness, but they remain vulnerable to adaptive attacks. Successfully attacking the latest, strongest models seems to require fairly long malicious prompt injections, and we take advantage of this in our design. 

Secure-by-design agent architectures. More recent work addresses prompt injection at the ar-chitectural level with privilege separation (Provos et al., 2003; Willison, 2023; Debenedetti et al., 2025; Beurer-Kellner et al., 2025; Costa et al., 2025; Jacob et al., 2026). These approaches sepa-rate processing of untrusted information from privileged execution, typically using the Dual LLM pattern (Willison, 2023) or multi-stage workflows. Existing methods instantiate this idea through plan-first execution, structured communication interfaces, restricted data types, or constrained infor-mation flow between trusted and untrusted contexts (Debenedetti et al., 2025; Beurer-Kellner et al., 2025; Costa et al., 2025; Jacob et al., 2026). These approaches significantly improve security, but often rely on heavily engineered workflows, rigid communication protocols, or task-specific security policies. This can lead to significant utility degradation and thus is not practical for all tasks.

## 4 APPLYING TWIN AGENT TO EXISTING AGENT SYSTEMS 

Twin Agent is designed as a general-purpose privilege separation framework that preserves the rea-soning dynamics of existing undefended agents with minimal task-specific redesign. To demonstrate this property, we instantiate Twin Agent on substantially different agent systems, including SWE-agent for SWE-bench (Jimenez et al., 2024; Yang et al., 2024) and tool-using agents for AgentDojo (Debenedetti et al., 2024) and DecodingTrust-Agent (Chen et al., 2026). These systems differ in reasoning styles, execution environments, tool interfaces, and threat surfaces, enabling us to evaluate the generality of Twin Agent across heterogeneous agent architectures. Across all instantiations, the core adaptation pattern remains consistent: (1) privileged actions are separated from untrusted information processing; (2) an Explore Agent is introduced to access the full untrusted context; and (3) the Explore Agent communicates compact hints to the Safe Agent, to try to keep the trajectory close to that of the original undefended agent. 4.1 SWE-AGENT FOR SWE-BENCH 

A SWE-bench (Jimenez et al., 2024) agent autonomously fixes GitHub issues by generating patches to real-world code repositories. User-submitted GitHub issues can be submitted by untrusted users, and hence could contain prompt injection attacks (Jacob et al., 2026). Therefore, a malicious issue could manipulate a naive bugfixing agent to make malicious changes to the repository, such as adding a backdoor to the code. To evaluate this risk, we create SWE-bench-injected , a modification of SWE-bench where issue descriptions are injected with attacks that instruct the agent to perform an attacker-specified action (i.e., writing to a file, deleting a file). The baseline SWE-agent is a ReAct-style software engineering agent designed for repository-level debugging and issue resolution (Yao et al., 2023; Yang et al., 2024). The original agent is provided with detailed instructions on how to read issue descriptions, inspect repository state, generate debugging hypotheses, and execute bash commands while observing the resulting execution output. We apply the Twin Agent methodology to this baseline agent to secure it against prompt injection. See the system prompts of different agents in Section D. The Explore Agent can read the full issue description and repository contents and optionally pass a hint to the Safe Agent. It also outputs a virtual bash command that would not be executed. The Safe Agent can read and write the repository and create a patch, but cannot read the issue description (which comes from an untrusted source). In each reasoning step, the Explore Agent optionally provides a compact hint to the Safe Agent, conditioned on the Safe Agent’s trajectory so far, to suggest how the Safe Agent can make progress. 6The Safe Agent can execute bash commands, and the resulting output is added to the shared context and can be observed by both agents. 4.2 TOOL -U SE AGENT FOR AGENT DOJO AND DECODING TRUST -AGENT 

We also instantiate Twin Agent on AgentDojo (Debenedetti et al., 2024) and DecodingTrust-agent (Chen et al., 2026), which evaluate LLM agents across diverse tool-use environments and multi-tool interaction settings. In both benchmarks, tool outputs are untrusted, while the system messages and the user’s instructions are trusted. The original undefended agent is designed to fulfill users’ requests by iteratively calling various tools and observing their output, creating a large attack surface for prompt injection. We apply Twin Agent to the baseline undefended agent. The Explore Agent can observe tool outputs and provide hints, while the Safe Agent performs privileged actions based only on trusted context and the hint messages. 4.3 WHY IS IT SECURE ?The security of Twin Agent is based on a key assumption: it is hard to mount successful prompt-injection attacks on frontier models, using short injections. We have observed that the security of the latest models is improving, and they can still be attacked, but the attacks tend to require long injections that eventually convince the model to follow the attacker instructions. Therefore, we limit the length of hints (and truncate or block hints that are longer than the limit), to prevent them from being used as an avenue of attack. To add security, we add an additional guardrail: we apply an off-the-shelf prompt injection detector to each hint, and block the hint if it is detected to contain a prompt injection. (Our detector uses a generic prompt found in Section A.) Therefore, we rely on the following assumption: it is hard to find a short injection that will simultaneously evade detection by our detector and also successfully attack the Safe Agent’s model. Our experiments with strong adaptive attacks appear to support this assumption. Because the hint is the only input to the Safe Agent that can be influenced by untrusted content, it is the only risk for prompt injection; by eliminating that risk, Twin Agent is secure-by-design against prompt injection.

## 6 CONCLUSION 

Prompt injection remains a significant barrier to the real-world deployment of agentic systems. We propose Twin Agent, a privilege-separated architecture that defends against prompt injection by adopting insights from compression. Instead of exposing a privileged agent to raw untrusted context or forcing a rigid plan-first workflow, Twin Agent separates exploration from execution and allows the Explore Agent to send only compact hints to the Safe Agent. Experiments on SWE-bench-injected, AgentDojo, and DecodingTrust-Agent show that this design preserves much of the original agent’s utility while substantially reducing attack success rates, outperforming both undefended agents and prior methods for system-level defense against prompt injection. Our analysis further shows that the security–utility tradeoff can be empirically controlled through the hint budget, suggesting that our approach is practical for building adaptive yet secure LLM agents. 

Limitations. Despite these results, Twin Agent still has several limitations. We have evaluated on only a few application domains; more work is needed to evaluate it on other tasks. Also, Twin Agent empirically appears secure, but it provides no formal guarantees, so it is conceivable that stronger attacks could defeat our design.

## B TRAJECTORY PRESERVATION ANALYSIS 

A key design goal of Twin Agent is to preserve the reasoning dynamics of the original utility-oriented agent while providing privilege separation. To evaluate whether Twin Agent preserves the original reasoning trajectory, we compare the trajectories produced by the defended agent with the trajectories of the unrestricted original agent on SWE-bench. We also consider a no-hint Safe Agent baseline, in which the Safe Agent cannot observe the untrusted region and receives no residual hints from the Explore Agent. We assess trajectory similarity at the semantic level using an LLM judge. For each task instance, we split the candidate trajectory into 10 checkpoints. The LLM judge outputs a trajectory deviation score from 0.0 to 1.0, where 0.0 indicates a trajectory nearly identical to the undefended agent, and 1.0

indicates a trajectory completely different from the undefended agent. Figure A2 visualizes some examples. Twin Agent achieves a substantially lower trajectory deviation from the undefended agent than the no-hint baseline, indicating that the hint messages help preserve the original dynamics.
